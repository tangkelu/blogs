# Numeric Parameters And Standardization Kickoff

Date: 2026-04-26

## Purpose

This kickoff resets the next phase of `llm_wiki` work away from supplier-lane hardening as the primary thread and toward the numeric-parameters / standardization main line.

The supplier-lane hardening wave is treated as workflow-control groundwork that is now sufficient for handoff, not as the main forward workstream.

The new main line is organized on three separate axes:

1. capability parameters
2. standards / qualification / acceptance handling
3. supplier-scoped dated records

This is a kickoff note, not a claim that numeric readiness has been unlocked.

Public reusable numeric readiness remains unchanged unless later work lands real primary-source support or dated supplier records that are acceptable under corpus policy.

## Why Shift Now

The current blocker pattern is no longer mainly about prose containment.

`20-layer` and `22-layer` are still blocked because the remaining gaps concentrate in three numeric authority layers:

- fabrication-capability numbers that public vocabulary pages do not authorize
- standards / qualification / acceptance thresholds that metadata-safe registry pages do not unlock
- supplier-specific capability / compliance / production numbers that require dated supplier evidence instead of generic public framing

The supplier-lane hardening pass was still necessary, but it now has diminishing returns as the primary workstream. It clarified the boundaries. It did not create reusable numbers.

So the shortest path is to build a numeric program that separates:

- what can become shared reusable capability parameters
- what can become shared reusable standardization cards
- what must remain supplier-scoped and dated by default

## Priority Queues

### Queue 1: `20-layer` and `22-layer`

These should be the first named targets because they are the clearest concentration of unresolved numeric blockers.

- `20-layer` is blocked mainly on geometry, process-window, HDI/build-up reliability, and HIL-specific capability numbers.
- `22-layer` is blocked mainly on standards thresholds, qualification / acceptance numerics, and HIL-specific compliance / approval assertions.

### Queue 2: Shared B-class capability numerics

This queue exists to recover reusable numeric classes that can later serve multiple layer-count articles instead of only one draft.

Priority shape:

- trace / space classes
- minimum drill classes
- minimum laser-via classes
- aspect-ratio classes
- annular-ring classes
- impedance-tolerance classes
- registration-tolerance classes
- board-thickness-window classes
- backdrill / residual-stub classes

These are shared only if they can be anchored to acceptable reusable authority. If not, they stay out of the reusable layer.

### Queue 3: Shared standards / standardization cards

This queue is for reusable standards-family structure and threshold handling, especially where multiple blogs will try to reconstruct the same acceptance language.

Priority shape:

- `IPC-601X` family and addendum hierarchy cards
- `IPC Class 2 / 3 / 3A` boundary cards
- qualification vs acceptance vs release-authority separation cards
- workflow vs threshold separation cards
- outgassing / screening / lot-conformance boundary cards where public metadata exists

The point is not to restate hidden thresholds from memory. The point is to standardize what the corpus can safely say and what it must still refuse.

## First Numeric Targets

The shortest path is to start with a small set of numeric classes that are both high-frequency and reusable across drafts.

### Shared reusable classes to attempt first

1. `trace/space`
2. `minimum mechanical drill`
3. `minimum laser via`
4. `aspect ratio`
5. `annular ring`
6. `impedance tolerance`
7. `registration tolerance`

This first wave is intentionally compact.

It is limited to the smallest cross-draft capability-parameter set that most directly affects whether future drafts can carry compact capability tables at all.

These are first because they recur across `20-layer`, `22-layer`, and adjacent high-layer drafts without automatically collapsing into supplier-proof or standards-threshold logic.

### Shared reusable but second-wave classes

- board thickness window
- backdrill residual stub
- plating thickness only if anchored to real threshold authority, not inferred summaries
- bow / twist only if anchored to real threshold authority, not metadata pages

These should wait until the first shared class policy is stable, because they more easily collapse into standards-threshold reconstruction.

### Classes that remain supplier-scoped unless stronger public authority appears

- production lead time
- quick-turn windows
- yield ranges
- cost multipliers
- lamination-cycle allowances used as actual factory capability proof
- HIL-specific `we can build X geometry` statements
- accepted qualification-package or approval-status claims

These should be handled only as supplier-scoped dated records, not as reusable public numeric facts.

## Standardization Targets

The standardization work should create a fixed handling model for standards-heavy drafts, especially `22-layer`.

This axis is not a capability-parameter queue.

It is a separate control layer for how standards, qualification, acceptance, and release language are normalized without pretending that hidden thresholds have been recovered.

### First standardization targets

1. Separate standards-family identity from threshold identity.
2. Separate method names from qualification or acceptance numerics.
3. Separate workflow vocabulary from accepted-result vocabulary.
4. Separate public standards metadata from supplier conformance assertions.
5. Separate reusable boundary cards from supplier-scoped dated compliance records.

### First card families to build or tighten

- class/addendum hierarchy cards
- clause-family vs threshold cards
- qualification vs listing vs release-authority cards
- acceptance-workflow vs acceptance-threshold cards
- compliance-assertion boundary cards

This is the minimum standardization layer needed before `22-layer` can move beyond containment.

## Supplier-Scoped Numeric Targets

Some numbers are structurally not public reusable until a supplier record exists.

That supplier-scoped lane should stay explicit and dated, not blended into shared capability-parameter cards or shared standardization cards.

### First supplier-scoped targets

- `20-layer` dated geometry/capability records for actual shop windows
- `20-layer` dated process-control or reliability records if they are meant to support capability claims
- `22-layer` dated compliance / qualification / acceptance-support records
- dated HIL-specific production, lead-time, and approval-status records where marketing language currently overreaches public support

### Minimum rule for supplier-scoped records

- supplier-specific
- dated
- parameter-specific
- non-transferable by default
- not promotable into public reusable readiness unless a separate acceptable source policy is satisfied

Interpretation rule:

- capability parameters, standards / qualification / acceptance handling, and supplier-scoped dated records are related but separate axes
- progress on one axis does not imply that the others are solved
- supplier-lane intake discipline does not by itself recover public reusable numerics

## Stop Conditions

Stop this kickoff phase if any of the following happens:

1. shared numeric cards start reconstructing numbers from memory, derivative tables, or metadata-safe pages instead of primary authority
2. standards cards start drifting from boundary control into invented threshold summaries
3. supplier-scoped records are treated as if they automatically unlock public reusable numeric readiness
4. `20-layer` or `22-layer` drafting starts claiming that numeric blockers are solved before the underlying source or dated-record layer exists

The default status at the end of this kickoff remains:

- public reusable numeric readiness: unchanged
- shared standards-threshold readiness: unchanged except for cleaner control structure
- supplier-specific numeric readiness: not unlocked until dated record intake lands

## Immediate Next Actions

1. Build a compact numeric-class queue for `20-layer` and `22-layer` using the existing blocker maps as the demand signal.
2. Start with first-wave shared B-class capability numerics: `trace/space`, drill, laser-via, aspect ratio, annular ring, impedance tolerance, registration tolerance.
3. In parallel, define the first standardization card set for `Class 3 / addendum / qualification / acceptance` boundary handling so `22-layer` stops trying to reconstruct thresholds.
4. Open a separate supplier-scoped dated-record intake lane for HIL-specific numeric parameters that cannot enter shared reusable cards.
5. Do not mark any blog or fact family as newly numeric-ready from this kickoff alone.

## Outcome For This Phase

The intended outcome is not `numbers recovered`.

The intended outcome is a disciplined queue that tells the program:

- which numeric classes are worth recovering first
- which ones can become shared reusable classes
- which standards areas need standardization before more drafting
- which numbers must remain supplier-scoped and dated

Until that work lands with acceptable evidence, `20-layer`, `22-layer`, and all adjacent high-density numeric reuse should continue to be treated as blocked for public reusable numeric claims.
