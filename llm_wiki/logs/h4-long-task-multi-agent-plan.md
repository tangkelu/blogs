# H4 Long-Task Multi-Agent Plan

Date: 2026-04-26

## Purpose

This control note turns `H4` from a kickoff plus queue cards into a durable multi-agent execution program.

It exists to:

- fix lane order and ownership
- keep `H4` from collapsing back into short single-note pushes
- ensure each lane produces execution-ready routing or control outputs instead of only another kickoff refresh

This file is not:

- a numeric recovery result
- a standards-threshold recovery result
- a supplier-evidence unlock
- proof that public reusable numeric readiness improved

### Controller Agent

Owns:

- lane order
- integration of subagent planning and outputs
- final wording in new `H4` notes
- updates to `backlog`, `phase-status`, and `update-log`
- stop/go decisions on whether a lane only improved routing or actually changed readiness

  - `annular ring`

- class-by-class control notes

### Lane Agent B: Standards / Standardization

Owns:

- `Class 3 / addendum / qualification / acceptance` standardization lane
- `22-layer` as the first named blocked branch

- one `22-layer`-anchored routing/control layer
- no threshold recovery by default

### Lane Agent C: Supplier-Scoped Dated Records

Owns:

- supplier-scoped dated numeric/parameter-record lane
- `20-layer` first

Current expected outputs:

- record-family queue notes
- narrow branch-specific control notes
- no promotion from supplier-scoped fact into shared reusable numerics by default

## Lane Output Contract

Every `H4` lane output should explicitly include:

1. purpose
2. current baseline
3. lane-specific inventory or routing surface
4. acceptable authority or policy posture
5. default refusal or hold posture
6. branch priority
7. completion criteria
8. explicit readiness statement

If no numeric or threshold recovery occurred, the file must say so directly.

## Lane Order

### Lane 1: Shared `B` Capability Parameters

This lane goes first because it is the shortest path toward `more numbers` without immediately collapsing into standards-threshold reconstruction or supplier-proof claims.

First-wave phase order:

1. `trace/space`
2. `minimum mechanical drill`
3. `minimum laser via`
4. `aspect ratio`
5. `registration tolerance`
6. `impedance tolerance`
7. `annular ring`

Parallel structure:

- Phase A:
  - `trace/space`
  - `minimum mechanical drill`
  - `minimum laser via`
- Phase B:
  - `aspect ratio`
  - `registration tolerance`
- Phase C:
  - `impedance tolerance`
- Phase D:
  - `annular ring`

Completion standard:

- each class lands a stable routing/control note
- each class ends at one of:
  - `refuse_numeric_recovery`
  - `keep_non_numeric_context_only`
  - `hold_for_supplier_evidence`
- no class is described as numerically recovered unless separate acceptable authority is later proven

### Lane 2: Standards / Standardization

This lane goes second, but may run in parallel with Lane 1 after controller kickoff because its write scope is separate.

First card order:

1. `standards-family identity vs threshold identity`
2. `qualification vs listing vs release-authority`
3. `acceptance-workflow vs acceptance-threshold`
4. `supplier-conformance assertion boundary`
5. `22-layer` branch routing/control mapping

Parallel structure:

- Group A:
  - `standards-family vs threshold`
  - `qualification vs listing vs release-authority`
- Group B:
  - `acceptance-workflow vs acceptance-threshold`
  - `supplier-conformance assertion boundary`
- Final integration:
  - `22-layer` branch routing/control mapping

Completion standard:

- four reusable separation cards exist
- one `22-layer`-anchored routing layer exists
- `20-layer` inheritance is explicit where useful
- no card behaves like a threshold substitute

### Lane 3: Supplier-Scoped Dated Records

This lane remains active but subordinate.

It should not reopen `H3` scaffolding work.

It should use the landed `H3` workflow-control substrate only to process narrow dated record families that cannot enter shared capability or shared standards lanes.

Recommended branch order:

1. `20-layer`
2. `22-layer`

Recommended first record families:

- `20-layer`
  - `supplier_capability_fact`
  - `supplier_process_control_fact`
  - `supplier_qualification_package_existence`
- `22-layer`
  - `supplier_status_or_listing_fact`
  - `lot_or_build_workflow_fact`
  - `qualification_or_compliance_package_existence`

Completion standard:

- each unit stays one branch, one record family, one dated candidate record, one bounded context
- each unit ends only at:
  - `reject_at_intake`
  - `hold_for_governed_review`
- no supplier-scoped fact is described as public reusable numeric recovery

## Tracking Rules

After each meaningful lane landing, sync all three:

- `llm_wiki/logs/backlog.md`
- `llm_wiki/logs/phase-status.md`
- `llm_wiki/logs/update-log.md`

Tracking language must state:

- which lane advanced
- what execution layer landed
- whether routing/control improved
- whether readiness changed
- what still remains blocked

## Stop Conditions

Stop and record `readiness unchanged` if the latest pass only does one or more of the following:

- splits a parameter class into narrower routing buckets
- splits a standards-heavy claim family into narrower identity routes
- improves refusal posture or blacklist precision
- improves supplier-record discipline without changing authority
- improves prompt-safe retrieval without stronger acceptable evidence

Do not record any of the following unless separately proven:

- shared capability numeric recovery
- standards-threshold recovery
- accepted-result or release-authority recovery
- supplier approval or conformance proof
- public reusable numeric readiness improvement

Lane-specific hard stops:

- do not let supplier-scoped proof become shared reusable numeric authority
- do not let standards hierarchy or workflow pages backdoor threshold values
- do not let workflow naming become accepted-result logic
- do not let listing ecosystem naming become current approved status
- do not let commercial figures become static evidence

## Current Decision

`H4` should now run as a controller-led multi-agent long task:

- Lane 1 pushes shared `B` capability-parameter routing first
- Lane 2 runs reusable standards / qualification / acceptance standardization in parallel
- Lane 3 remains a bounded supplier-scoped dated-record lane under the `H3` workflow substrate
- all three lanes remain active without any claim that public reusable numeric readiness has already improved
