## Policy Classes Used In This Matrix

The routing rules use the same three policy classes already established in the broader threshold/standardization program:

### 1. `metadata_only`

Allowed for:

- standards-family identity
- hierarchy identity
- addendum lineage
- workflow naming when it stays non-numeric
- ecosystem naming when it stays non-approval and non-release

Not allowed for:

- thresholds
- sample plans
- accepted-result proof
- supplier conformance proof
- release authority

### 2. `controlled_exclusion`

Default for:

- threshold identity without public primary threshold text
- acceptance-threshold implication
- qualification-package implication
- accepted-lot implication
- supplier-conformance implication
- release-authority implication

### 3. `supplier_scoped_dated_only`

Use only if a future claim needs:

- supplier-specific compliance proof
- dated listing or certification status
- release/approval authority evidence
- accepted qualification or acceptance records

This class does not promote content into reusable public readiness.

Public reusable numeric readiness remains unchanged even when this route is used.

## First Separation Axis: Standards-Family Identity Vs Threshold Identity

### Route A: Standards-family identity

Send claims here when they only say:

- a standard family exists
- an addendum branch exists
- a clause family exists
- a lineage or revision family exists
- a procurement or governance hierarchy exists

Default route:

- `metadata_only`

First cards to support this route:

- `Class / addendum hierarchy` cards
- `standards-family vs threshold` cards

Allowed output shape:

- `medical addendum exists`
- `automotive addendum exists`
- `space/military addendum exists`
- `military rigid-board family has a sheet-level branch`

Blocked output shape:

- `therefore the threshold is X`
- `therefore Class 3 requires Y here`
- `therefore the acceptance limit is now reusable`

This route is a boundary route, not threshold recovery.

Public reusable numeric readiness remains unchanged.

### Route B: Threshold identity

Send claims here when they attempt to recover:

- explicit limits
- acceptance counts
- sample-plan values
- clause-to-number mappings
- public reusable threshold tables

Default route:

- `controlled_exclusion`

Current execution rule:

- do not build threshold cards from metadata-safe material
- do not let clause-family visibility become pseudo-threshold recovery

This route stays blocked for `22-layer`.

This route does not become open just because the hierarchy is cleaner.

Public reusable numeric readiness remains unchanged.

## Second Separation Axis: Qualification Vs Listing Vs Release-Authority

### Route C: Qualification ecosystem identity

Send claims here when they only establish:

- qualification program existence
- listing ecosystem identity
- validation-service structure
- pre-acquisition listing/governance context

Default route:

- `metadata_only`

First cards to support this route:

- `qualification vs listing vs release-authority` cards

Allowed output shape:

- `a qualification listing ecosystem exists`
- `a listing layer can sit before acquisition-time acceptance`
- `a validation or audit ecosystem exists`

Blocked output shape:

- `supplier is currently qualified`
- `supplier is currently listed`
- `supplier is approved for this build`
- `supplier has release authority`

### Route D: Listing / approval / release-authority implication

Send claims here when they imply:

- current supplier qualification status
- current listing status
- customer approval
- accepted qualification package
- program release authority
- lot or build acceptance authority

Default route:

- `controlled_exclusion`

Upgrade route only if future evidence exists:

- `supplier_scoped_dated_only`

Execution rule:

- never infer this layer from public standards-family or listing metadata

This route is a hard blocker for `22-layer`.

Public reusable numeric readiness remains unchanged.

Default route:

- `controlled_exclusion`

Execution rule:

- no numeric or accepted-result unlock is created by workflow naming alone

This route is especially important for the `22-layer` branch and remains blocked.

Public reusable numeric readiness remains unchanged.

## Fourth Separation Axis: Supplier-Conformance Assertion Boundary

### Route G: Standards metadata kept as metadata

Send claims here when they remain at:

- document identity
- ecosystem naming
- process/governance vocabulary
- hierarchy or lineage framing

Default route:

- `metadata_only`

Required wording bias:

- `public metadata identifies`
- `programs may require`
- `can involve`

### Route H: Supplier conformance assertion

Send claims here when they imply:

- supplier conforms to the standard
- supplier can release to the standard
- supplier can accept lots to the standard
- supplier has validated qualification standing
- supplier marketing is being upgraded into standards proof

Default route:

- `controlled_exclusion`

Upgrade route only if future evidence exists:

- `supplier_scoped_dated_only`

Execution rule:

- public hierarchy metadata, workflow vocabulary, and listing ecosystem pages do not cross this boundary by themselves

This is the direct containment route for supplier-proof drift in `22-layer`.
