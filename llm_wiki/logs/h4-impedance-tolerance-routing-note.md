## Purpose

This file is the `H4` Lane 1 Phase C single-parameter routing note for `impedance tolerance`.

It exists to standardize how `impedance tolerance` claims should be routed before any attempt is made to recover a shared reusable numeric field.

This file is:

- a routing/control note
- a single-parameter blocker map
- a demand-branch execution note for Lane 1 Phase C

- a numeric recovery result
- a recovered capability table

- proof that public reusable numeric readiness improved

## Current Blocker Shape

The current blocker shape for `impedance tolerance` is stable and unchanged:

- the corpus has impedance-related overview context, but not acceptable shared reusable numeric authority
- `impedance tolerance` is often used as a compressed proxy for fabrication control, verification workflow, SI confidence, and shop capability rather than as a directly authorized reusable parameter
- overview pages about controlled impedance can describe process context, but they do not authorize a transferable shared tolerance number
- standards metadata and standards-family visibility can describe related governance context, but they do not publish a safe shared reusable `impedance tolerance` number for this repo
- supplier marketing and supplier-scoped capability claims can mention tolerance posture, but those claims remain supplier-scoped and non-transferable by default
- derivative summaries and heuristic tables often present tolerance ranges as if they were universal defaults without acceptable source authority

The key split that must remain intact:

- `impedance tolerance` is a fabrication capability parameter, not a standards-threshold surrogate
- it is separate from verification or reporting workflow naming
- it is separate from SI-outcome implication
- it is separate from supplier-scoped dated proof

## Acceptable Authority Requirement

`Impedance tolerance` should be treated as shared reusable only if acceptable authority actually exists.

Minimum requirement:

- a primary source must expose the impedance-tolerance field directly
- the scope must be explicit enough to show whether the tolerance is shared reusable, supplier-scoped, process-scoped, product-scoped, or build-context-specific
- the source must distinguish design-target identity, fabrication-capability identity, and verification/reporting identity
- the source must make reuse conditions explicit rather than forcing reconstruction from overview prose, standards metadata, or marketing language

Not acceptable as shared reusable numeric authority:

- impedance-related overview pages
- standards metadata
- supplier marketing pages
- supplier-scoped proof pages
- dated supplier capability records used outside their supplier scope
- derivative comparison tables
- generic overview pages without direct parameter authority

If a future source is supplier-scoped and dated, it may support a narrow supplier-only handling lane.

It still does not become shared reusable numeric authority by default.

## Default Refusal Posture

If acceptable authority is missing, default posture is:

- refuse the exact `impedance tolerance` number
- keep only non-numeric impedance-control or verification-process context
- reject overview, standards-metadata, supplier-marketing, and supplier-proof materials as numeric basis
- preserve separation between capability routing, verification workflow, and SI-outcome implication
- route any supplier-specific tolerance claim away from shared reusable handling

Default refusal wording should bias toward:

- `impedance-tolerance capability depends on source-specific scope`
- `public reusable impedance-tolerance authority is not yet established`
- `non-numeric impedance-control context may be kept, but the numeric tolerance remains blocked`

This note fixes the non-reuse boundary for `impedance tolerance`.

The following do not qualify as shared reusable numeric authority:

- impedance-related overview pages

These sources may still help with:

- vocabulary control
- claim splitting

They do not authorize a shared reusable impedance-tolerance number.

## Routing Use Rule

When a draft asks for `impedance tolerance`, the controller should decide only:

- which branch is creating demand
- whether the claim is really a capability-tolerance claim rather than a verification, standards, or supplier-proof claim
- whether acceptable direct authority exists
- whether the claim must remain non-numeric or be routed to a narrower supplier-scoped lane

This note is routing only.

It is not numeric recovery.
