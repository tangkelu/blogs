# H2 Backdrill And Stub Source Map

Date: 2026-04-25

## Purpose

This file maps the current local corpus into support classes for the `backdrill_and_stub` H2 bucket.

It is a governance and source-mapping document.

It is not:

- a source registry record
- a fact card
- approval to publish transferable backdrill, residual-stub, oversize, or depth-control numbers

## Relation To H2 Policy

Under [h2-capability-number-policy.md](/code/blogs/llm_wiki/logs/h2-capability-number-policy.md), backdrill depth control and residual-stub windows are fabrication-capability numerics.

That means:

- internal process or product pages can support framing and recovery routing
- method and process pages can support posture, workflow, and non-claim boundaries
- none of the current internal prose sources, by themselves, authorize a reusable transferable backdrill number
- future `keep` status still requires a `Tier 1 internal dated capability record` with scope and refresh discipline

### 2. Posture Support

These sources are the strongest current support for how backdrill should be described as an engineering control.

They support posture and workflow language, not reusable backdrill numbers.

#### [backdrill-control-capability.md](/code/blogs/llm_wiki/facts/methods/backdrill-control-capability.md)

  - posture support

  - explicitly normalizes that internal pages present backdrill as standard high-speed and RF support discipline
  - already states that exact residual-stub limits, oversize rules, and verification criteria require current engineering confirmation
- what it supports safely:
  - backdrill is part of normal transition-control posture for certain high-speed, RF, and backplane contexts

  - backdrill matters more in dense multilayer and long-through-via contexts

## What Is Still Missing

The missing item is not more backdrill vocabulary. The missing item is a governed numeric authority.

Still missing:

- a registered `Tier 1 internal dated capability record` for backdrill depth control
- a registered `Tier 1 internal dated capability record` for residual-stub targets, if those are to be reused at all
- explicit separation between backdrill capability, residual-stub targets, and depth / oversize / verification controls
- explicit separation between fabrication-control numbers and SI / performance numerics
- usable date, owner, scope, and refresh posture for any future backdrill-control record
- evidence that a cited number is a maintained capability record rather than a service-page residue or engineering shorthand

Without those items, all reusable backdrill and stub numerics remain `needs_source`.

1. Recover or register the missing `Tier 1 internal dated capability record` for backdrill control.
2. Verify that the record cleanly separates `backdrill capability`, `residual stub target`, and `depth / oversize / verification control`.
3. Verify that SI / RF outcome numerics are excluded from the capability record rather than embedded inside it.
4. Only after that, use the current internal registry pages as framing cross-checks for family names, use cases, and downstream blog routing.
5. Re-run the H0/H2 demand split so each blocked claim is sorted into `backdrill capability`, `residual stub target`, `depth / oversize control`, or `SI / performance numeric`.
6. Only after steps `1-5` should any `backdrill_and_stub` fact card be considered.
