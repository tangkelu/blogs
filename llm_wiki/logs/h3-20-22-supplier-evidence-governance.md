# H3 20-Layer And 22-Layer Supplier-Evidence Governance

Date: 2026-04-26

## Purpose

This note creates a dedicated governance layer for future supplier-evidence work on the two still-blocked `H3` branches.

It exists to:

- define what claim classes are permanently parked in the supplier-evidence lane
- define what claim classes might be unlockable later if dated supplier or lot records ever appear
- define what claim classes must never be genericized into reusable numerics even if supplier records exist

This file is not:

- a supplier-evidence batch
- a readiness unlock
- permission to reopen `P4-06` for `20-layer` or `22-layer`

## Current Governance Baseline

Current shared reality:

- `20-layer` and `22-layer` are both control-hardened and live-draft-contained
- both branches remain blocked for high-density numeric reuse
- public-primary-source work is now narrowed to metadata tightening only
- any future unlock attempt must pass through a separate supplier-evidence discipline

This note therefore governs future supplier evidence before any such evidence exists.

## Governance Split

Future supplier-evidence handling for these two branches must distinguish three classes:

1. `permanently_parked_supplier_only`
2. `narrow_unlockable_if_dated_supplier_record_exists`
3. `never_genericize_even_if_supplier_record_exists`

The point of this split is to stop two failure modes:

- treating supplier facts as public reusable numeric rules
- treating dated supplier or lot events as timeless board capability

## Branch A: 20-Layer

### 1. Permanently Parked In Supplier-Evidence-Only

The following `20-layer` claim classes are permanently parked outside public-source recovery:

- HIL-specific fabrication capability claims
- HIL-specific process-control numerics
- HIL-specific production-scale, yield, lead-time, and cost claims
- geometry and stack-recipe claims when used as HIL capability proof
- finished-board qualification or accepted-reliability proof

These classes may only be revisited inside a future supplier-evidence discipline.

### 2. Narrowly Unlockable If Dated Supplier Records Exist

The following `20-layer` classes may be unlockable later, but only in narrow supplier-scoped form:

1. dated supplier-specific capability or process facts
2. dated supplier-specific qualification or reliability-package existence facts
3. dated supplier-specific geometry or recipe facts
4. dated supplier-specific operational or delivery facts

Rule:

- these are unlockable only as dated, scoped, supplier-bound facts
- they do not become generic `20-layer` truths

### 3. Never Genericize Even If Supplier Records Exist

The following `20-layer` classes must never be promoted into reusable public numerics:

- geometry and feature-size tables
- via, drill, aspect-ratio, registration, or impedance tolerance tables
- stack-recipe and lamination-count defaults
- process-window and control-plan numerics
- `IST`, coupon, sample-plan, and accepted-reliability thresholds
- yield, lead-time, cost, and production-scale numerics

Rule:

- supplier evidence may support a dated supplier profile
- it must not create a generic `20-layer` rulebook

## Branch B: 22-Layer

### 1. Permanently Parked In Supplier-Evidence-Only

The following `22-layer` claim classes are permanently parked outside public-source recovery:

- HIL-specific `Class 3 / 3A` conformance claims
- HIL-specific qualification-package or acceptance-package claims
- HIL-specific accepted-status, approved-source, trusted-source, certification-status, or release-authority claims
- HIL-specific `FAI`, coupon, microsection, traceability, `SPC`, controlled-environment, or operator-certification wording used as proof
- HIL-specific sector-readiness or compliance claims
- HIL-specific lot-acceptance or lot-release proof claims

These classes may only be revisited inside a future supplier/lot-evidence discipline.

### 2. Narrowly Unlockable If Dated Supplier Or Lot Records Exist

The following `22-layer` classes may be unlockable later, but only in narrow date-bound form:

1. dated supplier status, approval, listing, or certification facts
2. dated lot-, build-, or first-article-specific workflow facts
3. dated qualification-package or compliance-package existence facts
4. dated accepted-lot or release-event facts
5. dated program-specific readiness statements with explicit scope

Rule:

- these are unlockable only as dated, scoped, customer/program/lot-bound facts
- they do not become generic `22-layer` conformance or acceptance truth

### 3. Never Genericize Even If Supplier Or Lot Records Exist

The following `22-layer` classes must never be promoted into reusable public numerics or reusable proof logic:

- `Class 3 / 3A` thresholds
- addendum technical criteria
- acceptance tables
- `PLT` sample-plan and lot-conformance numerics
- outgassing, screening, or qualification acceptance numerics
- `FAI`, coupon, or workflow converted into generic acceptance-path logic
- traceability, `SPC`, controlled environment, or operator-certification converted into generic conformance proof
- any dated supplier status rewritten as timeless `HILPCB complies with` language

Rule:

- supplier or lot records may support narrow factual event statements
- they must not become public reusable standards logic

## Record Requirements For Any Future Unlock Attempt

Any future supplier-evidence unlock attempt on these two branches must provide:

1. a date
2. a named scope
3. a named site, supplier, lot, build, or program context
4. an explicit statement that the fact is supplier-scoped rather than public-general
5. an explicit statement that the fact does not override current `H2` / `H3` governance on reusable numerics

If any of the above is missing, the claim stays parked.

## Rejection Rules

Reject any future supplier-evidence proposal if it tries to:

- turn one supplier record into a generic layer-count rule
- turn a dated event into a timeless capability claim
- turn qualification workflow into universal pass/fail authority
- turn a supplier status into public standards conformance for all contexts
- route commercial or operational facts into stable evidence-pack numerics

## Tracking Interpretation

If a future supplier-evidence batch lands, tracking must distinguish:

- `supplier-scoped factual support improved`
- `public reusable numeric readiness unchanged`

Do not record:

- `20-layer unlocked`
- `22-layer unlocked`
- `high-density numeric ready`

unless a later dedicated readiness note explicitly proves that outcome.

## Recommended Next Execution Order

1. treat this note as a parking and unlock-governance layer only
2. do not start a supplier-evidence batch unless there is a real dated-record path
3. if supplier evidence does appear later, recover only the narrowest supplier-scoped facts first
4. keep all generic thresholds, geometry tables, process windows, and commercial numerics outside reusable evidence packs

## Completion Criteria

This governance note is complete when:

1. both blocked branches have explicit parked claim classes
2. both blocked branches have explicit narrow-unlock classes
3. both blocked branches have explicit never-genericize classes
4. tracking docs can treat any future supplier evidence as supplier-scoped rather than public-general by default

## Minimal Control Posture

For `20-layer` and `22-layer`, future supplier evidence may at most support narrow, dated, scoped supplier or lot facts.

It does not by itself authorize:

- reusable public numerics
- reusable standards thresholds
- generic capability tables
- generic acceptance or qualification logic
