# H4 Trace Space Routing Note

Date: 2026-04-26

## Purpose

This note is the first single-class routing note under `H4` Lane 1 Phase A.

It exists to convert the shared first-wave `trace/space` class from queue-level routing into a narrower controller-owned class note.

This file is:

- a single-class routing note
- a shared `B` capability-parameter control note
- a refusal/hold posture note for future evidence handling

This file is not:

- a numeric recovery result
- a recovered geometry table
- permission to write reusable `trace/space` numerics into evidence packs or blog drafts
- proof that public reusable numeric readiness improved

Public reusable numeric readiness remains unchanged after this note.

## First Named Blocked Branches

The first named blocked branches for this class remain:

- `20-layer`
- `22-layer`

They stay first because `trace/space` pressure most often arrives there as a compressed high-density capability claim, not as acceptable reusable parameter authority.

Secondary downstream demand may later appear in:

- `14-layer`
- `18-layer`
- `24-layer`

That downstream reuse is hypothetical only.

This note does not unlock it.

## Main Demand Branches

Current highest-demand branches for this class are:

- `20-layer`
- `22-layer`
- `14-layer`
- `18-layer`

Why demand recurs:

- drafts repeatedly use `trace/space` as shorthand for density
- density shorthand is often rewritten as shop-capability proof
- the same shorthand can drift toward standards-threshold or acceptance implication even when no such authority exists

## Current Blocker Shape

The current blocker shape for `trace/space` is stable and does not depend on one draft only.

The main problems are:

1. generic high-density manufacturability wording gets compressed into unsupported geometry minima
2. supplier-side capability posture gets misread as transferable shared numeric authority
3. standards-heavy or inspection-heavy context gets allowed to strengthen geometry pressure even when it does not publish reusable `trace/space` values
4. branch demand gets mistaken for evidence

So the current obstacle is not only missing numbers.

It is claim-family collapse around what kind of authority a `trace/space` number would actually require.

## Acceptable Authority Requirement

No reusable `trace/space` numeric should be written unless a primary authority exposes the field directly with clear scope and condition.

Minimum acceptable authority rule:

- the source must expose `trace/space` directly as a parameter field
- the source must make scope explicit enough to tell whether the parameter is:
  - product-scoped
  - process-scoped
  - lot- or build-scoped
  - supplier-scoped
- the source must be stronger than generic process overview, marketing posture, standards hierarchy, or draft demand

Additional rule:

- if the only available authority is supplier-scoped, the result must remain supplier-scoped and dated
- supplier-scoped authority does not automatically become a shared reusable class

## Default Refusal Posture

If the authority above is missing, the controller should default to:

- `refuse_numeric_recovery`

Allowed fallback posture:

- `keep_non_numeric_context_only`

Supplier-trigger posture:

- `hold_for_supplier_evidence`

Interpretation:

- refuse reusable `trace/space` numerics
- keep only non-numeric density or manufacturability context
- route supplier-specific geometry or capability claims into the supplier-scoped dated-record lane rather than the shared `B` lane

## Non-Reuse Boundary

The following do not qualify by default as reusable `trace/space` authority:

- supplier-scoped proof pages
- factory marketing summaries
- standards-threshold pages
- TOCs, clause-family summaries, or hierarchy pages
- derivative comparison tables
- broad process-overview pages
- draft demand from `20-layer`, `22-layer`, or adjacent branches

These source shapes may help boundary control.

They do not become reusable shared numeric basis for `trace/space`.

## Minimal Current Verdict

The correct current routing verdict for this class is:

- `refuse_numeric_recovery` by default
- `keep_non_numeric_context_only` where guarded density/manufacturability context is still useful
- `hold_for_supplier_evidence` where the claim is actually supplier-scoped capability

This note improves class-level routing only.

It does not recover a reusable `trace/space` number.

Public reusable numeric readiness remains unchanged.
