# H2 Registration Source Map

Date: 2026-04-25

## Purpose

This file maps the current corpus for the `registration` bucket into support classes before any numeric recovery attempt.

It is a source-governance map, not a fact card and not a source record.

## Relation To H2 Policy

Under `h2-capability-number-policy.md`, registration is a fabrication-capability numeric.

That means:

- internal process or product pages can support posture and recovery routing
- high-layer and hybrid-stackup pages can support boundary control
- none of the current internal prose sources, by themselves, authorize a reusable transferable registration number
- future `keep` status still requires a dated capability record with scope and refresh discipline

## Missing Items

Missing items before `registration` can move beyond intake:

- a dated internal capability record for registration capability
- explicit measurement basis for any future registration claim
- explicit separation between:
  - rigid multilayer registration capability
  - rigid-flex registration posture
  - standards/class claims
  - warpage / flatness / SI-adjacent claims
- explicit site, line, or process-family scope where needed
- explicit refresh rule and supersession chain
- one intake note confirming that no class threshold, warpage number, or SI claim was imported through this bucket

## Recommended Ingestion Order

1. internal high-layer posture inputs
   - start with `frontendapt-pcb-high-layer-count-pcb-page-en`
2. rigid multilayer scope inputs
   - then `frontendapt-pcb-multilayer-pcb-page-en` and `frontendhil-multilayer-pcb-product-page-en`
3. normalized posture layer
   - then `facts/methods/high-layer-count-backdrill-and-registration-posture.md`
4. public/context boundary inputs
   - then `facts/methods/high-layer-rigid-board-manufacturability-context.md`
5. lamination and hybrid boundary inputs
   - then `frontendapt-pcb-multi-layer-laminated-structure-page-en` and `wiki/processes/hybrid-rf-stackup-strategy.md`
6. rigid-flex boundary inputs
   - then `frontendapt-pcb-rigid-flex-pcb-page-en`, `frontendapt-rigid-flex-pcb-capability-page-en`, `frontendhil-rigid-flex-pcb-product-page-en`
7. only after that, decide whether a real dated capability record can be created

If step 7 fails, the bucket remains:

- `needs_source` for reusable numerics
- `downgrade` for non-numeric manufacturability wording

## Current Source-Map Conclusion

The current corpus is strong enough to map `registration` boundaries and to prevent collapse between registration capability, process posture, standards/class language, and warpage/SI-adjacent claims.

It is not strong enough to recover a reusable registration number today.
