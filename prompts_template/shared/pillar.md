# Pillar Prompt

你要输出一篇可直接发布的中文 PCB / PCBA 技术博客，文章类型为 `Pillar`，围绕主关键词：`{{keyword}}`。

这篇文章的目标是：

- 系统覆盖一个宽主题，形成可持续扩展的主题中心页
- 覆盖定义、边界、设计、制造、验证、选型、误区和下一步
- 帮助搜索系统和 AI 系统识别这个站点在该主题上的专题权威

你会收到或预先准备以下输入：

- 主关键词：`{{keyword}}`
- 日期：`{{date}}`
- 标签：`{{tags}}`
- evidence pack：包含公开事实、项目级判断、不可写边界
- 目标站点 overlay：HILPCB 或 APTPCB
- 必要时加载的模块：`comparison`、`application`、`capability`

严格执行以下要求。

## 输出要求

1. 只输出最终 Markdown。
2. 首字符必须是 `---`。
3. 不输出分析过程、推理过程、提示词解释或写作说明。
4. 不写泛介绍，不写空转背景，不写 AI 口吻。
5. 文章要像资深工程团队为工程师、采购、项目团队写的体系化技术指南。
6. 这篇页面必须体现出 hub 页角色，而不是单点问答页。
7. 结构完整不代表专题结论自动成立；凡是标题、description、目录或 H2 承诺的标准、参数、频段、材料、方法、应用边界，都必须在正文里被兑现并被正确支撑。

## 数据边界

### 可以写成公开事实

- 标准组织、联盟、监管机构、官方 datasheet、白皮书、公开规范中的事实
- 公开可验证的定义、术语、测试方法、材料属性、工艺原理、验证框架

### 只能写成项目级判断

- DFM 冻结项
- 量产评审优先级
- 试产检查项
- 选型与 trade-off 建议
- 什么时候该优先看设计、制造、装配、测试或供应商约束

### 禁止写入

- 编造工厂能力、客户结果、认证、案例数字
- 编造行业统一数值
- 把特定站点公开能力写成行业通用规则

如果 evidence pack 没有提供可验证数字，就不要补数字；改写成影响因素、判断条件或验证动作。

## 博客重写 / 生成前置契约

在生成正文前，必须先按 `blog-rewrite-data-gap-contract.md` 执行一次前置判断。该判断是硬门槛，不是建议。

默认流程：

1. 先拆解博客或关键词的关键点：主题边界、系统性覆盖范围、工程决策、必须解释的概念、需要的总览表 / 检查表 / 选择框架、站点承接动作、不可写 claim。
2. 检查 `llm_wiki` 是否已有足够 source / fact / wiki / gate 支撑顶尖专题页。
3. 如果状态是 `safe_but_generic`、`needs_data` 或 `hold`，不能直接写正文；必须先补 `llm_wiki` 或缩小标题、description、目录和 H2 承诺。
4. 只有状态达到 `ready`，或数据缺口已补齐并形成 evidence pack 后，才能继续执行正文生成。
5. 重写时必须消费新增数据，不能补了数据但正文仍停留在泛泛解释。

禁止把“结构完整、没有明显错误”当作顶尖稿合格标准。Pillar 页面必须形成专题级判断框架，而不是把多个泛化 Query 段落堆在一起。

## Evidence Pack 强制执行

在生成正文前，必须按 `evidence-pack-consumption-contract.md` 执行一次 evidence pack 消费检查。该检查是硬门槛，不是建议。

### 状态处理

| Evidence status | 允许写法 | 禁止写法 |
| --- | --- | --- |
| `verified` | 可写成公开事实；数值只能在该 fact 自身已验证且不属于 hard refusal class 时使用 | 不得扩展成 pack 未授权的相邻参数、能力表或阈值 |
| `framing_only` | 只能写成定性背景、专题边界、风险语境、流程判断或项目级建议 | 不得升级成数字、标准阈值、能力上限、验收结论或供应商证明 |
| `blocked` | 只能作为删除、拒写或内部 `DATA_GAP` 信号 | 不得进入正文、FAQ、表格、标题、description 或目录承诺 |
| `must_refresh` | 刷新前视为不可用 | 不得写入正文、表格、FAQ 或来源区 |
| `supplier_scoped_dated_only` | 只能在保留供应商、日期和适用范围的专门语境中使用 | 不得提升为共享行业事实、HILPCB/ATPPCB 通用能力或可迁移供应商证明 |

### Hard Refusal Classes

以下 claim classes 默认禁止写入，除非当前 evidence pack 明确给出更窄的 `verified` 支撑，并且该支撑没有被标成 `supplier_scoped_dated_only`：

- shared `B` capability numerics：`trace/space`、最小机械钻孔、最小激光孔、aspect ratio、annular ring、impedance tolerance、registration tolerance 及类似 capability table 数字
- standards thresholds：`IPC Class 2 / 3 / 3A`、`IPC-6012 / 6013`、`IST`、`PLT`、coupon plan、plating minima、bow/twist limits、acceptance threshold 及类似验收阈值
- supplier proof：supplier approval、qualification、compliance、release authority、accepted-lot status、可迁移工厂能力证明
- process windows / recipes：lamination-count allowance、stack recipe、bake / press cycle、build-up default、manufacturability window
- high-speed / RF budget numerics：channel-loss budget、insertion-loss table、Nyquist budget、RF geometry rule、性能预算表
- dynamic commercial numerics：lead time、quick-turn window、yield、cost uplift、MOQ、区域或客户商业数字

### 缺证据时的强制行为

如果某个专题结论、总览表、FAQ、标题承诺、目录项或模块需要 `verified` 支撑但 evidence pack 没有提供，必须按顺序处理：

1. 如果已有 `framing_only` 支撑，降级为定性框架、验证动作或项目级建议。
2. 如果无法安全降级，删除该句子、表格行、FAQ、目录项或模块。
3. 如果缺口会影响 hub 页系统性，在工作检查中标记 `DATA_GAP`，并缩小标题、description、目录或 H2 承诺。

禁止用相邻事实、行业常识、标准名称、供应商页面、workflow 名称或旧稿数字补齐缺口。

### 内部控制信息不得进入正文

Evidence pack 里的 `Disposition`、`Prompt action`、`Current-blog section bridge map`、`Unsupported / Unclear`、`Not cleared`、`do not write`、`downgrade`、`delete`、`blocked`、`DATA_GAP` 等内容只能用于生成前判断，不能原样或改写后进入最终文章。

最终正文应把这些控制信息转换为读者可用的工程表达：

- 把“不要怎么写 / 不应写什么”改成“项目风险信号 / 需要确认什么 / 适用边界是什么”
- 把“删除某类数字”改成“该参数需按具体 stackup、材料和 DFM 确认”
- 把“blocked claim”改成不出现该 claim，或改成面向项目的验证动作
- 表格列名不得出现 `不应怎么写`、`禁止写入`、`Prompt action`、`Disposition`、`Evidence status`、`DATA_GAP` 等内部审稿语义
- H2 / H3 不得暴露“模板、审稿、策略、证据包、生成规则”这类内部流程
- 作者与审核信息不得出现 `llm_wiki`、内部文件路径、内部数据层、内部审核 workflow 或任何仓库/提示词系统名称；应改成“工程内容审核团队”“制造工程团队”“材料与工艺审核团队”等对外可理解实体

在开始写正文前，默认再做一次快速自检：

- 这个 pillar 真正覆盖的边界是什么
- 标题、description、目录承诺了哪些主题
- 哪些 claim 属于公开事实
- 哪些 claim 只能写成项目级判断
- 哪些章节必须放正文内联来源归因
- 哪些标准号、标准版本、日期、频段、材料家族、测试方法或合规名词必须再核对一次
- 哪些 claim 的 evidence status 不是 `verified`，需要降级、删除或标记 `DATA_GAP`
- 是否存在 hard refusal class 被写入标题、description、目录、表格、FAQ 或正文
- 是否把 evidence pack 的内部控制语句误转成了正文栏目、表格列名或小标题
- 作者与审核信息是否完全移除了 `llm_wiki`、内部路径、内部 workflow 和提示词系统名称

## 文章必须包含的结构

- frontmatter
- 唯一 H1 大标题
- 核心结论
- 一个可被 AI 单独摘录的定义型摘要 / quick definition block
- 定义与边界
- 至少一张早期总览表
- 第二层早期结构化信息块
- 设计 / 制造 / 验证 / 选型 / 误区中的核心章节
- FAQ
- 公开参考资料
- 与站点解决路径相关的下一步
- 目标站点 overlay 要求的内置表单、询盘或 RFQ 组件
- 作者与审核信息

## Frontmatter

必须输出如下结构，并替换占位符：

```yaml
---
title: "<主关键词前置的体系化标题>"
description: "一句话说明本文系统覆盖该主题的定义、设计、制造、验证与选型。"
category: technology
date: "{{date}}"
featured: true
image: ""
readingTime: 10
tags: {{tags}}
---
```

## H1 规则

- frontmatter 之后必须紧跟一个且仅一个正文 H1，大标题内容默认与 frontmatter.title 保持一致或高度一致
- frontmatter.title 不能替代正文可见 H1
- H1 必须位于正文前部，且全文不要再出现第二个 H1
- 如果页面使用首图，允许顺序为 `frontmatter -> 首图 -> H1 -> 核心结论`
- 如果没有首图，默认顺序为 `frontmatter -> H1 -> 核心结论`

## 标题规则

- 标题必须把主关键词尽量前置。
- 标题要体现体系化、专题化、指南化，但不要写得像百科词条。
- 优先使用“完整指南”“工程指南”“设计与制造指南”“从定义到验证”这类表达。
- 标题不要为了覆盖宽词而承诺正文实际没有系统展开的子主题。
- 如果标题或 description 写了 `design`、`manufacturing`、`validation`、`cost`、`lead time`、`materials`、`standards`、`FR1/FR2`、`reliability` 这类强 promise，正文必须给出对应的判断框架、边界和来源支撑。

## 正文结构

### 1. 核心结论

紧跟 frontmatter 输出。

要求：

- 用 `5-7` 条短结论概括整篇文章
- 这些结论要覆盖定义边界、关键风险、关键选择、验证路径和应用价值
- 读者只看这一段，也应能抓住主题的核心判断

### 1.5 定义型摘要 / Quick Definition Block

在 `核心结论` 之后，必须增加一个适合 AI 抽取的短定义块。

要求：

- 用 `40-60` 词左右概括这个主题“是什么 / 为什么重要 / 最先看什么”
- 必须能脱离全文上下文独立成立
- 语气要像专题页定义，不要写成营销标语
- 如果定义块里出现标准、频段、材料、测试、参数或公开技术结论，必须确认这些说法在正文和来源中都能落住

### 2. 目录

Pillar 页面默认启用目录。

要求：

- 目录放在 `核心结论` 与定义型摘要之后、正文主体之前
- 目录优先对应核心 H2、FAQ、结语、作者与审核信息
- 目录项要便于人和 AI 快速理解页面结构，不要机械复制超长标题

### 3. 什么是 `{{keyword}}`，它和相邻概念有什么区别？

要求：

- 明确定义边界
- 说明它不等于什么
- 解释它在工程、采购、量产或可靠性上为什么重要

### 4. 总览表：工程上先看哪些维度？

必须尽早出现，优先使用以下列结构：

`维度 | 为什么重要 | 常见取值或影响因素 | 怎么验证或确认`

如果主题带有明显比较意图，可改成更强的决策矩阵。

### 5. 第二层早期结构化信息块

默认保留。如果主题属于以下任一类型，必须强化这一层：

- 材料
- 阻抗
- 叠层
- 可靠性
- 测试
- 工艺控制
- 高频高速
- 热管理

使用以下二选一：

- 4-card 风格信息块
- 多张补充 Markdown table

这一层要尽早表达：

- 关键 trade-off
- 设计与制造耦合点
- 验证路径
- 常见误判

默认判断规则：

- 对阻抗、材料、叠层、可靠性、高速、工艺控制这类参数密集型主题，优先考虑 `总览表 + 4-card HTML 卡片`
- 对成本驱动、优先级排序、对比矩阵、方案评估这类主题，如果补充表格更清楚，就优先 `总览表 + 多张 Markdown table`

HTML 卡片使用规则：

- 卡片只能浓缩正文里已经成立的工程结论、风险信号、验证动作、冻结项或选型逻辑
- 不要把卡片写成抽象口号或装饰性文案
- 卡片不是每篇都必须有，但在一个专题集群中也不要长期退化成纯表格
- 如果同一专题的多篇文章都使用卡片，不要机械复用同一套配色、渐变和视觉氛围

HTML 卡片配色规则：

- 配色按主题走，不按随机审美走
- 材料 / 表面处理 / 制程类：偏材料感与工艺感配色
- 功率 / 热 / 高电流类：偏工业能量与热路径配色
- 医疗 / 低噪声 / 高可靠性类：偏洁净、克制、低饱和配色
- 高频高速 / SI / 阻抗类：偏冷静、技术感、传输路径感配色
- 变化要有规律，既不能整批一模一样，也不能每篇像不同模板系统
- 所有卡片都必须保持高对比度、强可读性和工程文档气质

### 6. 核心章节

以下章节至少覆盖其中四个；如果主题天然适配，优先全覆盖：

- 设计视角：真正决定成败的关键点
- 制造与装配视角：从图纸到量产会遇到什么问题
- 如何选型：什么时候该用，什么时候不该用
- 如何验证：测试、检查与评审路径
- 成本 / 交期驱动因素
- 应用场景差异
- 失效模式与常见误区
- 标准与合规边界

每个主要 H2 必须遵守：

- 开头先给结论
- 再展开范围、trade-off、风险和验证
- 如果该段包含具体数值、标准目标、材料参数、验证方法或公开结论，正文内必须加入自然的内联来源归因，例如“根据 IPC 规范”“Rogers datasheet 给出”“某 layout guide 指出”
- 如果该段提到标准号、标准版本、频段定义、材料家族、协议版本或测试方法编号，写入前必须先核对是否准确、是否为当前有效口径
- 尽量用表格、列表、对比结构，而不是长段背景介绍

### 7. 常见误区

必须单独成节。

要求：

- 指出工程上最常见的错误理解
- 说明为什么错
- 给出更稳妥的判断方式

### 8. 与站点解决路径相关的下一步

要求：

- 把专题理解自然连接到具体产品、服务、材料、能力、工具或 RFQ
- 优先导向目标站点的商业承接页，博客页为次级补充
- 内链要分散在页面前部、中部、后部，不要过度集中
- 锚文本要自然嵌入，不做生硬 SEO 堆砌
- 如果目标站点 overlay 定义了内置询盘、报价或 RFQ 组件，正文必须按 overlay 要求插入，不能只在文字里写“联系我们”或只放普通链接

### 9. 常见问题（FAQ）

必须保留以下包裹结构：

```html
<!-- faq:start -->
<!-- faq:end -->
```

FAQ 要求：

- 覆盖正文未完全展开、但搜索和咨询中高频出现的问题
- 问题要能独立被检索
- 问题优先贴近自然语言搜索问句和 query phrasing，而不是纯目录式措辞
- 回答要直接，不复制整段正文

### 10. 可选：术语表

只有在术语密度高、且确实会降低阅读门槛时才添加。

### 11. 结语

用一小段总结：

- 这个主题为什么值得作为长期工程能力建设项
- 什么时候应继续深入到具体 query 文章、材料选择、stackup 确认、DFM 或报价动作

### 12. 作者与审核信息

必须包含：

- 作者角色
- 技术审核角色
- 最近更新语义

优先要求：

- 如果站点具备真实作者、审核人、工程负责人或专题负责人实体，应优先使用真实可识别实体
- 如果暂时无法给出真实姓名，至少给出更明确的职能范围、审核责任和团队边界
- 作者与审核信息要让页面具备专题权威信号，而不是匿名营销页

### 13. 公开参考资料

正文结尾必须增加 `公开参考资料` 或 `Sources` 区块。

要求：

- 至少列出 3 条公开可点击来源
- 优先使用标准组织、官方 datasheet、官方设计指南、白皮书、测试方法说明、官方技术文章
- 每条来源都要说明它支撑了正文中的哪个关键主题或判断
- 不能只裸贴链接，必须便于人工和 AI 系统理解来源用途
- 如果正文提到了标准 identity、版本号、频段边界、方法编号或材料参数，参考资料区必须能回溯到这些 claim，而不是只给泛化二级资料

## AI 可引用性要求

- 这篇 hub 页必须能被 AI 系统抽取出定义、维度、决策框架、FAQ 和下一步路径
- 关键结论不能只在文末找来源；涉及公开结论的正文段落应有自然的内联来源归因
- 页面前 30% 内容必须让 AI 和读者都能快速识别主题边界、关键变量和验证路径
- 优先使用总览表、决策矩阵、定义块、FAQ、短列表，不要把所有信息埋进长段落
- 页面要表现专题权威，而不是泛泛百科摘要

不要伪造真实姓名；如果没有明确姓名，用角色表述。

## 模块加载规则

### comparison 模块

当 pillar 本身就是 `X vs Y`、`A or B`、`difference between` 类型专题时，补充：

- 早期比较矩阵
- 选择逻辑
- 例外边界

### application 模块

当主题具有明显行业应用属性时，补充：

- 行业 / 场景表
- 场景差异化约束
- 对应验证动作

不要把“应用”写成泛行业罗列，要写成不同终端需求下的设计、材料、工艺、测试差异。

### capability 模块

当主题天然连接制造能力、RFQ、交期、MOQ 或服务选择时，补充：

- 已公开 capability 快照
- 工程文件、DFM、打样或批量导入所需资料
- 从专题理解走向商务动作的自然承接

限制：

- 只能使用站点已公开能力
- capability 内容不能淹没专题主体

## 写作规则

- 这是一篇 hub 页，必须表现出“总览 + 决策框架 + 下一步分流”能力。
- 每个 H2 都要 answer-first。
- 优先使用表格、列表、短段落、对比结构。
- 对参数密集型主题，尽量在前 30% 内容中交代关键变量和验证路径。
- 如果某个说法无法确认来源，就改写成“项目中通常需要确认”“应按供应商能力与验证结果判断”。
- 如果某个标题词、目录项或 H2 承诺的范围过大，而正文证据和展开不够，应主动收缩承诺，不要用宽词硬撑专题权威。
- 不要强制加入 glossary、supplier checklist、cheat sheet、capability 大表、行业应用大表，除非主题确实需要。
- 不要泄露“模块”“模板”“提示词”“overlay”“evidence pack”等内部术语。

## 质量门槛

最终文章必须同时满足：

- 能被读者视为该主题的总览页，而不是普通问答页
- 能让工程师快速建立该主题的决策框架
- 能让采购或项目团队理解哪些约束会影响报价、交期、可制造性和验证
- 能让搜索系统和 AI 系统抽取定义、维度、规则、FAQ 和下一步路径
- 能自然分发到目标站点的商业页面和后续子主题页面
- 关键专题结论在事实边界上站得住，不因为“像完整指南”就放松标准、版本、参数和来源核对

## 最终发布前自检

在输出最终 Markdown 前，默认逐条确认：

- 标题、description、目录和关键 H2 承诺的范围，正文是否真正系统覆盖
- 是否存在“宽词进、窄词出”或“专题标题很大、正文只覆盖其中一部分”
- 正文关键专题结论是否有对应的正文内联来源归因
- 文末参考是否真的能回溯到正文中的标准、版本、参数、测试方法、材料或合规结论
- 是否把项目级判断误写成行业统一硬规则
- 是否存在标准号、版本号、日期、频段定义、材料家族名、测试方法编号写错或过时
- 是否因为缺公开数据而仍然硬补了数字、阈值或 capability
- 是否已经按 `verified / framing_only / blocked / must_refresh / supplier_scoped_dated_only` 完成 evidence pack 状态处理
- 是否已经删除或降级所有 hard refusal class，而不是把它们藏进总览表、FAQ、卡片或 capability 模块
- 如果某个 claim 无法核实，是否已降级成条件判断、验证动作或项目级建议
- 目标站点 overlay 要求的内置表单、询盘或 RFQ 组件是否已经插入到正文合适位置，且没有放进 frontmatter、代码块、表格或 FAQ 包裹中
