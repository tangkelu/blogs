# Query Prompt

你要输出一篇可直接发布的中文 PCB / PCBA 技术博客，文章类型为 `Query`，围绕主关键词：`{{keyword}}`。

这篇文章的目标是：

- 直接回答一个明确的工程问题、制造问题、选型问题、成本问题、交期问题或排查问题
- 帮助读者完成设计、选型、制造、装配、验证或供应商沟通决策
- 让内容更容易被搜索系统、AI 搜索和问答系统抽取

你会收到或预先准备以下输入：

- 主关键词：`{{keyword}}`
- 日期：`{{date}}`
- 标签：`{{tags}}`
- evidence pack：包含公开事实、项目级判断、不可写边界
- 目标站点 overlay：HILPCB 或 APTPCB
- 必要时加载的模块：`comparison`、`application`、`capability`

严格执行以下要求。

## 输出要求

1. 只输出最终 Markdown。
2. 首字符必须是 `---`。
3. 不输出分析过程、推理过程、提示词解释或写作说明。
4. 不写 AI 口吻，不写“本文将”“接下来我们”之类空泛导语。
5. 语言要像工程团队或制造团队写给工程师、采购、项目经理看的技术文档。
6. 先回答问题，再解释背景，再展开细节。

## 数据边界

### 可以写成公开事实

- 标准组织、行业联盟、监管机构、官方 datasheet、白皮书、官方工艺说明中的事实
- 公开可验证的定义、术语、测试方法、合规框架、材料属性、公开工艺原理

### 只能写成项目级判断

- DFM 优先级
- 试产或量产中的风险排序
- 什么时候更适合选 A 或 B
- 检查项、验证路径、控制带、评审优先级
- 供应商沟通重点、报价所需资料、试产建议

### 禁止写入

- 编造数字、阈值、良率、交期、案例结果
- 编造工厂 capability、认证、设备、产能、客户结果
- 把项目经验写成行业统一标准
- 把站点产品页描述包装成全行业事实

如果 evidence pack 没有给出数字，就不要补数字；改写成条件判断、影响因素或验证方法。

## 文章必须包含的结构

- frontmatter
- 唯一 H1 大标题
- H1 下方顶部答案块
- 一个可被 AI 单独摘录的定义型摘要 / quick answer block
- 至少一张早期规则 / 参数 / 对比表
- 2-4 个围绕关键词意图展开的主体 H2
- FAQ
- 公开参考资料
- 与站点解决路径相关的下一步
- 作者与审核信息

## Frontmatter

必须输出如下结构，并替换占位符：

```yaml
---
title: "<主关键词前置的查询式标题>"
description: "一句话回答搜索意图，并说明本文覆盖哪些规则、风险、验证方法。"
category: technology
date: "{{date}}"
featured: true
image: ""
readingTime: 8
tags: {{tags}}
---
```

## H1 规则

- frontmatter 之后必须紧跟一个且仅一个正文 H1，大标题内容默认与 frontmatter.title 保持一致或高度一致
- 不要把 frontmatter.title 当作可见 H1 的替代
- H1 必须出现在正文前部，不能缺失，也不要在后文再出现第二个 H1
- 如果有首图，允许顺序为 `frontmatter -> 首图 -> H1 -> 顶部答案块`
- 如果没有首图，默认顺序为 `frontmatter -> H1 -> 顶部答案块`

## 标题规则

- 标题必须把主关键词尽量前置。
- 标题要匹配查询意图，优先使用 `What Is`、`How to`、`X vs Y`、`Cost`、`Lead Time`、`Troubleshooting`、`Design Rules` 一类表达。
- 不写空泛品牌标题，不写新闻标题，不写夸张标题党。

## 正文结构

### 1. 顶部答案块

紧跟 H1 输出。

要求：

- 用 `4-6` 条短结论直接回答核心问题
- 每条结论都要可执行、可判断或可验证
- 这一段必须足够独立，单独被截取时也能回答问题
- 默认直接以短结论列表输出，不强制增加 `## 直接回答` 这样的可见小标题
- 如果主题确实需要过渡性小标题，可以使用更自然的工程标题，但不要机械重复“直接回答”模板字样

### 1.5 定义型摘要 / Quick Answer Block

在 `顶部答案块` 之后，必须增加一个适合 AI 抽取的短摘要块。

要求：

- 用 `40-60` 词左右的单段或单卡片直接定义核心问题
- 必须能脱离上下文独立成立
- 优先覆盖“它是什么 / 何时需要 / 最关键判断依据”
- 不要写空泛品牌话术，不要重复整段前言

### 2. 目录

长文默认启用目录。

要求：

- 当全文存在 `3` 个及以上主体 H2，或正文已经明显超过短答文章长度时，必须添加目录
- 目录必须放在 `顶部答案块` 与定义型摘要块之后、正文主体之前
- 目录项优先对应主体 H2、FAQ、结语、作者与审核信息等主要锚点
- 目录文本要可读，不要机械重复完整长标题

### 3. `{{keyword}}` 在工程上先看什么？

要求：

- 用一段话定义问题边界
- 明确说明它不等于什么，避免概念混淆
- 尽快告诉读者本问题最先看的 `3-5` 个维度

### 4. 关键规则与参数总表

必须尽早出现，优先使用以下列结构：

`规则 / 参数 | 推荐范围或判断方式 | 为什么重要 | 怎么验证 | 如果忽略会怎样`

如果主题是比较型，可改成：

`维度 | A 的表现 | B 的表现 | 什么时候更适合`

### 5. 第二层早期结构化信息块

如果主题属于以下任一类型，默认启用：

- SI
- DFM
- 材料
- 阻抗
- 背钻
- 可靠性
- EMC
- 连接器
- 容差控制
- 工艺窗口

这一层必须紧跟在第一张早期表格之后，使用以下二选一：

- 4-card 风格的信息块
- 额外一张或多张 Markdown table

这一层的目标是尽早呈现：

- trade-off
- 工程约束
- 失效风险
- 验证路径

### 6. 主体 H2

从以下方向中按关键词意图选择 `2-4` 个，不要全部堆上：

- 设计 / layout 关键点
- 制造 / stackup / 工艺关键点
- 装配 / 测试 / 检验关键点
- 失效模式与排查
- how to choose / trade-off
- cost / lead time 驱动因素
- 应用场景差异
- 供应商确认项

每个主要 H2 必须遵守：

- 开头先给结论
- 再解释工程原因
- 再解释制造影响或商业影响
- 最后给验证方法、确认动作或沟通动作
- 如果该段包含具体数值、规范目标、材料参数、测试方法或典型结论，正文内必须加入自然的内联来源归因，例如“根据 USB-IF 规范”“NXP layout guide 指出”“TI 应用笔记给出”

不要写成纯科普段落，也不要只堆概念。

### 6.5 可选增强：HTML 卡片

当主题存在以下任一情况时，可以插入 `1` 组轻量 HTML 卡片：

- 需要在前 30% 内容里快速呈现 trade-off
- 需要把“适用场景 / 主要风险 / 推荐动作 / 验证路径”并列展示
- 纯 Markdown table 不够直观，但仍然需要保持专业感

对以下参数密集型或工程控制型主题，默认优先考虑 `早期 Markdown table + 4-card HTML 卡片`：

- SI
- DFM
- 材料
- 阻抗
- 背钻
- 可靠性
- EMC
- 连接器
- 容差控制
- 工艺窗口

如果主题本身更适合用多张表表达，则不要强行插入卡片，优先保留 `早期 Markdown table + 补充 Markdown table`。这类主题通常包括：

- 成本优化
- 优先级排序
- trade-off 决策矩阵
- make vs buy 判断
- 纯比较型 / 排名型内容

卡片使用规则：

- HTML 卡片是增强项，不是每篇强制项
- 单篇建议最多 `1` 组，避免模板味过重
- 卡片必须总结正文里已经成立的工程信息，不能为了凑视觉块再发明阈值
- 卡片文案必须表达工程结论、冻结项、风险信号、验证动作或选型提醒，不能写装饰性营销文案
- 卡片不是随机装饰；只有在确实能提升扫描效率时才使用
- 即使使用 HTML 卡片，也不能替代关键规则表和正文主体

卡片视觉规则：

- 如果同一关键词集群内多篇文章都使用卡片，不要机械复用同一套主色、渐变和视觉气质
- 配色必须服从主题逻辑，而不是随机换色
- 可参考以下方向：
  - 材料 / 表面处理 / 制程对比：偏材料感、工艺感、树脂 / 铜 / 表面处理色系
  - 功率电子 / 逆变器 / 高电流：偏工业能量、热管理、深色工业控制色系
  - 医疗 / 低噪声 / 高可靠性：偏洁净、克制、低饱和、医疗感色系
  - 高频高速 / 阻抗 / SI：偏信号完整性、层叠、传输路径感的冷静技术色系
- 视觉变化要克制，目标是“同一系统下的分组变化”，不是每篇乱换，也不是每篇都一样
- 无论使用什么配色，都要优先保证文字对比度、扫描效率和工程感

### 7. 与站点解决路径相关的下一步

要求：

- 把技术判断自然连接到产品、服务、材料、能力、工具或 RFQ 动作
- 优先导向目标站点的产品页或服务页，博客页只能作为次级补充
- 内链分布要均衡，不要全部堆在开头或结尾
- 内链锚文本要语义自然，不要机械精确匹配堆砌

### 8. 常见问题（FAQ）

必须保留以下包裹结构：

```html
<!-- faq:start -->
<!-- faq:end -->
```

FAQ 要求：

- 只放真实会被问到的补充问题
- 问题要短、准、可检索
- 问题优先使用自然语言搜索问句和检索式问法，贴近真实 query phrasing，而不是目录式标题
- 回答要直接，不要重复正文整段内容

### 9. 结语

用一小段总结：

- 什么时候该重点关注这个问题
- 什么时候需要进一步做 DFM、stackup、材料或装配确认

### 9.5 公开参考资料

正文结尾必须增加 `公开参考资料` 或 `Sources` 区块。

要求：

- 至少列出 `3` 条公开可点击来源
- 至少列出 3 条公开可点击来源
- 优先使用标准组织、官方 datasheet、官方设计指南、白皮书、官方技术文章
- 每条来源都要说明它支撑了正文中的哪个结论
- 不要只裸贴链接，格式要适合人工审查
- 如果正文写了具体数值、阻抗目标、材料参数或测试方法，参考资料区必须能回溯到这些结论

### 10. 作者与审核信息

必须包含：

- 作者角色
- 技术审核角色
- 最近更新语义

优先要求：

- 如果站点有真实作者、审核人、工程负责人或技术团队实体页，优先使用真实可识别实体
- 如果暂时没有真实姓名，至少给出更具体的职能描述、团队边界和审核责任
- 作者与审核信息要让读者和 AI 系统理解这不是匿名营销文

## AI 可引用性要求

- 关键定义、规则、对比结论要能被 AI 直接摘录，不依赖上下文补全
- 正文不能只在文末列来源；关键结论所在段落应带内联来源归因
- 能写成结构化块的内容，优先写成表格、短列表、定义块、FAQ，而不是长段说明
- 如果主题涉及公开标准、datasheet、白皮书、layout guide、测试方法，必须显式体现出处类型
- 页面必须同时服务人类读者和 AI 系统，不要为了 SEO 牺牲可读性

不要伪造真实姓名；如果没有明确姓名，用角色表述。

## 模块加载规则

### comparison 模块

当关键词属于 `X vs Y`、`difference between`、`which is better`、`how to choose A or B` 时，必须补充：

- 一张早期比较表
- 明确决策规则
- 明确例外边界

不能只写“看需求而定”，必须说明什么需求对应什么选择。

### application 模块

当关键词具有明显行业 / 终端应用属性时，补充：

- 行业 / 场景表
- 场景差异化约束
- 场景下的验证动作

不要把行业应用写成营销案例，要写成设计约束和制造要求差异。

### capability 模块

当关键词具有强商业 / 制造 / 报价 / MOQ / lead time 意图时，补充：

- 已公开 capability 快照
- 交期、MOQ、RFQ 或 DFM 所需资料
- 从技术问题走向询盘动作的自然过渡

限制：

- 只能使用站点已公开能力
- capability 内容不能压倒技术主体

## 写作规则

- 每个 H2 都要 answer-first。
- 优先使用表格、列表、短段落，不要写成大段密集长文。
- 对复杂主题，优先把参数、约束、判断条件提前，而不是埋在后文。
- 如果某个说法无法确认来源，就改写成“项目中通常需要确认”“应按供应商能力和验证结果判断”。
- 不要强制加入 glossary、supplier checklist、cheat sheet、行业应用表、capability 大表，除非关键词真的需要。
- 不要泄露“模块”“模板”“提示词”“overlay”“evidence pack”等内部术语。

## 质量门槛

最终文章必须同时满足：

- 能让工程师在前 20% 内容里抓到核心判断
- 能让采购或项目经理看出该补什么资料、该确认什么风险
- 能让搜索系统抽取出定义、规则、参数、比较结论、FAQ
- 能自然承接到目标站点的商业页面
