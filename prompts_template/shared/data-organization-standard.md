# Data Organization Standard

本文件定义技术博客写作中“数据”该如何准备、分类、存储和调用。

目标不是让每篇都堆很多数字，而是让每篇文章都能明确：

- 哪些内容是公开可验证事实
- 哪些内容是项目级判断
- 哪些内容不能写
- 数据怎么进入模板，而不是在模板里硬编

## 一、为什么必须单独定义数据组织

技术博客最大的风险不是结构差，而是：

- 数据不足
- 数字来历不清
- 把经验判断写成行业标准
- 把站点能力写成普遍事实
- 同一主题反复临时搜资料，口径不统一

所以模板体系必须和数据体系拆开：

- 模板解决结构问题
- 数据组织解决可信度问题

## 二、四层数据模型

### 1. Public Facts

公开可验证事实，包括：

- IPC / IEC / UL / JEDEC 等标准语境
- 官方 datasheet
- 官方白皮书
- 联盟、监管机构、标准组织的公开说明
- 材料厂商公开参数

允许写法：

- 作为事实陈述
- 作为表格参数来源
- 作为验证方法依据

### 2. Engineering Judgments

工程判断，包括：

- DFM 优先级
- 设计权衡
- 风险排序
- 打样与量产阶段的控制建议
- 什么时候选 A，什么时候选 B

允许写法：

- 必须写成项目级判断
- 必须体现适用条件
- 应当说明这是经验性建议，而不是统一行业硬标准

### 3. Site-Specific Capabilities

站点公开能力，包括：

- 层数范围
- 材料范围
- 最小线宽线距
- 最小孔径
- 厚度范围
- lead time 范围

允许写法：

- 只能在 capability 模块或承接段落中使用
- 必须与站点公开页面一致
- 不能伪装成“行业普遍值”

### 4. Unsupported Claims

不能确认来源的说法，包括：

- 无出处数字
- 无出处良率 / 成本 / 可靠性百分比
- 未公开的客户案例数据
- 未公开的工厂能力
- 不确定的标准限值

处理方式：

- 删除
- 降级为定性判断
- 标记为待确认

## 三、每篇文章都应先准备 evidence pack

写作前至少要有一个轻量 evidence pack。

建议使用：

- `shared/evidence-pack-template.md`

每篇 evidence pack 至少包含：

- topic
- keyword
- 公开事实清单
- 项目级判断清单
- 不支持内容清单
- 来源类型
- 站点承接路径

## 四、数据采集优先顺序

### 第一优先级：标准与官方来源

- 标准组织
- 官方 datasheet
- 官方白皮书
- 官方材料手册

### 第二优先级：行业头部技术内容

可用于学习结构与问题框架，但不要直接复制内容。

主要用途：

- 找出高频问题
- 找出常见比较维度
- 找出常见表格结构
- 找出 FAQ 真实提问方式

### 第三优先级：站点自身公开页面

主要用途：

- 商业承接
- capability 模块
- 内链落点
- 与产品 / 服务 / 工具页的一致口径

## 五、数据在文章里的落点

不同类型的数据，应该进入不同位置。

### Public Facts 最适合放在

- 定义段
- 规则 / 参数表
- 标准 / 参考章节
- 验证方法章节
- FAQ

### Engineering Judgments 最适合放在

- 设计取舍
- 风险排序
- implementation steps
- troubleshooting
- how to choose

### Site-Specific Capabilities 最适合放在

- capability 模块
- RFQ / DFM 段落
- CTA 前后的承接段

### Unsupported Claims 不应进入正文

如果必须保留原意，就改写为：

- “需结合具体 stackup / 材料 / 板厚确认”
- “建议在 DFM / quote 阶段确认”
- “不同工厂和项目窗口会有差异”

## 六、数据表建议字段

如果后续要做统一数据表，建议至少包含：

| Field | Meaning |
| --- | --- |
| topic_id | 主题 ID |
| keyword | 关键词 |
| claim | 具体陈述 |
| claim_type | public_fact / engineering_judgment / site_capability / unsupported |
| source_type | standard / datasheet / whitepaper / official_site / internal_public_page |
| source_name | 来源名称 |
| source_url | 来源链接或路径 |
| confidence | high / medium / low |
| safe_wording | 安全写法 |
| notes | 备注 |

## 七、数字写法标准

### 可以直接写数字的情况

- 来自公开标准或 datasheet
- 来自站点公开能力页
- 来自文章已明确引用的公开来源

### 不应直接写数字的情况

- 没有确认来源
- 不确定是否为项目窗口值还是普遍能力
- 同行业不同厂差异很大

### 降级写法

把硬数字改为：

- “通常取决于材料体系、铜厚和层叠结构”
- “应以具体 stackup 计算和 DFM 确认为准”
- “量产窗口需结合公差与检测方法共同确认”

## 八、如何让“数据不足”不再成为问题

如果发现基于 `seo_rewrite` 和 `llm rewrite` 的稿子“数据很少”，处理方法不是继续加强模板，而是加强文章前置的数据准备。

建议固定流程：

1. 先做 evidence pack
2. 把事实、判断、能力、不可写内容分开
3. 先决定文章需要哪些表格
4. 再决定哪些数据能支撑这些表格
5. 数据不够时，宁可减少结构，也不要编表

## 九、表格和数据的关系

不要为了模板而硬放表格。

### 该放表格的情况

- 读者需要做选择
- 参数存在对比关系
- 验证动作和风险可以结构化
- 不同行业 / 场景的约束明显不同

### 不该强放表格的情况

- 没有足够可验证参数
- 只是泛泛介绍概念
- 实际上更适合用步骤或 FAQ 表达

## 十、推荐的证据增强优先级

对技术博客而言，优先补这些数据：

1. 关键定义与边界
2. 规则 / 参数 / 公差语境
3. 为什么重要
4. 怎么验证
5. 失效模式与排查逻辑
6. 什么时候该选 A，什么时候该选 B

比起堆很多数字，更重要的是让每个结论后面都有：

- 依据
- 适用条件
- 验证路径

## 十一、与模板的关系

今后的模板不应该再内置大量“假定已经有数据”的强制栏目。

正确关系是：

- `technical-blog-standard.md` 定义写作底线
- `template-selection-and-pruning.md` 定义模板和模块怎么选
- `keyword-cluster-design-standard.md` 定义关键词怎么成组
- `data-organization-standard.md` 定义证据和数据怎么准备
- `query.md` 与 `pillar.md` 只负责骨架

这样以后即使换站点、换主题、换模型，底层方法也不会乱。
